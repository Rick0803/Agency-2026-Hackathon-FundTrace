Here's the honest tradeoff map:

Option A: BigQuery (cloud computation)
What it would take:

Export all 4 large tables from PostgreSQL → CSV/Parquet
Upload to Google Cloud Storage
Load into BigQuery
Rewrite the query() layer in retrieval.py to use google-cloud-bigquery instead of psycopg2
Handle auth, billing, schema differences
Reality check for a hackathon: The data dump alone takes time, and the organizers' shared DB may block bulk COPY exports. BigQuery would be fast (sub-second for these CTEs), but the setup cost is hours, not minutes.

Option B: DuckDB in-memory
What it would take:

On startup, dump the 4 PostgreSQL tables into pandas DataFrames (~1–2 GB in RAM)
Register them with an in-memory DuckDB engine
Run the same SQL against DuckDB (milliseconds instead of seconds)
Problem: The slow part is fetching from PostgreSQL. Loading all of fed.grants_contributions (1.275M rows) into memory is itself a 30–60s operation, and you still need to do it at least once — same as the preload approach but with more memory pressure.

Option C: Background preload (module-level Python dict + thread)
What it would take:

~30 lines of code in a new preload.py, ~10 lines in app.py, ~15 lines in fetch.py
No new infrastructure
No auth changes
How it works: A daemon thread fires the two slow queries the instant the app process starts. By demo time (~60s after launch), results are already in memory. Button click is instant.

Recommendation
Option C is the right call for demo day. The bottleneck is the PostgreSQL query itself — BigQuery and DuckDB don't eliminate that first fetch, they just reformat where it happens. Preloading solves the only thing that actually matters: the delay the audience sees.

If you want BigQuery for the long-term product beyond the hackathon, that's worth building — but not in the next few hours.
